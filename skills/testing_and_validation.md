# Skill: testing_and_validation

## 目的
确保本项目每次改动都经过最低限度且有针对性的验证，而不是“看起来差不多能跑”。

## 使用时机
- 改动了解析规则时
- 改动了分类逻辑时
- 改动了 Excel 写入逻辑时
- 提交代码前

## 基本要求
1. 新增功能必须配套新增测试；修复 bug 必须补一个能覆盖该 bug 的测试。
2. 纯 UI 改动至少要做一次手动主流程验证。
3. 解析、分类、Excel 写入这三类核心逻辑优先使用自动化测试覆盖。
4. 如果当前环境无法完整运行测试，要明确记录未验证项和原因。

## 本项目建议测试层次
- `parser`：验证公司、岗位名、地点、关键词、技能、学历等提取结果。
- `classifier`：验证岗位类别、匹配方向、优先级、是否建议单独改简历。
- `excel_writer`：验证是否能写入 `JobBoard`，以及标题映射和追加逻辑是否正确。
- `app`：至少做一次手工链路验证，确认表单可编辑、可触发写入。

## 验证完成标准
- 关键测试通过
- 主链路能说明“输入什么、输出什么、写到哪里”
- 异常路径有清晰报错，不静默失败
